# Этапы разработки Telegram-бота EnergyCFO

Документ описывает практический план разработки MVP Telegram-бота **EnergyCFO | Встречи**. Его задача - разложить работу на понятные этапы, по которым можно идти последовательно: от подготовки аккаунтов и сервера до тестирования, запуска и развития.

Основа документа: [ТЗ_Telegram_bot_EnergyCFO_v1.4.md](ТЗ_Telegram_bot_EnergyCFO_v1.4.md).

## 0. Общая логика разработки

Проект делаем не сразу "большим приложением", а последовательными рабочими блоками:

1. Сначала подготавливаем учетные записи, домен, сервер, секреты и репозиторий.
2. Затем создаем минимальный каркас backend и Telegram-бота.
3. После этого добавляем базу данных и хранение заявок.
4. Затем реализуем пользовательский сценарий записи.
5. После пользовательского сценария добавляем админ-согласование.
6. Далее подключаем Google Calendar и Google Meet.
7. Потом добавляем бизнес-правила: лаг 12 часов, буфер 30 минут, лимит 5 встреч, исключенные даты.
8. После этого делаем напоминания, обработку ошибок, логи и резервные копии.
9. В конце проводим приемочное тестирование, выкладку и запуск.

Ключевой принцип: на каждом этапе должен появляться проверяемый результат, а не только "часть кода".

## 18. Этап наблюдения после запуска

### Цель

Первые дни после запуска контролировать реальные ошибки и быстро дорабатывать сценарии.

### Период наблюдения

Рекомендуемый период: 7-14 календарных дней после запуска.

### Что отслеживаем

| Показатель | Что делать при проблеме |
|---|---|
| Ошибки Telegram | смотреть логи и webhook |
| Ошибки Google Calendar | проверять OAuth и лимиты |
| Непонятные действия пользователей | улучшать тексты и кнопки |
| Зависшие заявки | проверять SLA-уведомления |
| Заполнение диска | чистить логи, проверить бэкапы |
| Неверные слоты | проверять алгоритм доступности |

### Результат этапа

Через 1-2 недели формируется список доработок v1.1:

| Возможная доработка | Основание |
|---|---|
| Google Sheets-отчет | нужны управленческие выгрузки |
| Шаблоны отказов | часто повторяются причины |
| Улучшение текстов | пользователи задают одинаковые вопросы |
| Mini App | кнопочный выбор слотов становится неудобным |
| Веб-админка | админ-настройки в Telegram становятся тесными |

## 20. Рабочий порядок для совместной разработки

Чтобы мы могли делать проект вместе, удобный порядок такой:

3. Создать репозиторий проекта.
4. Поднять минимальный backend.
5. Подключить Telegram webhook.

7. Подключить PostgreSQL.
8. Создать сущности пользователей и заявок.
9. Реализовать пользовательскую заявку без Google Calendar.
10. Реализовать админ-согласование без Google Calendar.
11. Подключить Google Calendar.
12. Добавить расчет слотов.
13. Добавить создание событий и Google Meet.
14. Добавить переносы, отмены, напоминания.
15. Провести приемочное тестирование.
16. Развернуть production.

## 21. Definition of Done для MVP

MVP считается готовым, если выполнены все условия:

| Блок | Критерий |
|---|---|
| Пользователь | может создать заявку от начала до конца |
| Администратор | может согласовать, отказать, предложить другое время |
| Google Calendar | событие создается, переносится и отменяется |
| Google Meet | ссылка создается автоматически |
| Слоты | учитывают все ограничения |
| Исключенные даты | работают на 3 месяца вперед |
| Напоминания | работают для администратора и пользователя |
| Ошибки | логируются и отправляются администратору |
| Бэкапы | настроены и проверены |
| Production | бот работает через `https://meet.energycfo.pro` |
| Приемка | UAT-сценарии пройдены |
