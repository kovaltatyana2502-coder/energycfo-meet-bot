# Промпт для разработки Telegram-бота EnergyCFO

Используй этот промпт как стартовую инструкцию для работы над проектом Telegram-бота **EnergyCFO | Встречи**.

```markdown
Контекст: работаем в текущей директории проекта Telegram-бота EnergyCFO.

Цель проекта: разработать Telegram-бота для согласования B2B-встреч с Татьяной Коваль. Пользователь выбирает тему, оставляет данные, выбирает свободный слот, а встреча создается в Google Calendar только после ручного согласования администратором.

## Файлы проекта

Перед стартом работы определи и проверь наличие ключевых файлов:

1. `ТЗ_Telegram_bot_EnergyCFO_v1.4.md`
   - основной источник продуктовых, бизнес- и функциональных требований;
   - использовать для сценариев, ролей, бизнес-правил, интеграций, критериев приемки;
   - не изменять без отдельного запроса;
   - если потребуется новая редакция ТЗ, создавать отдельный файл новой версии, например `ТЗ_Telegram_bot_EnergyCFO_v1.5.md`.

2. `Стек_технологий_Telegram_bot_EnergyCFO.md`
   - источник по утвержденному стеку, архитектуре и техническим решениям;
   - использовать для выбора технологий, структуры backend, БД, webhook, Docker, Nginx, SSL, Google API;
   - не менять стек без явного обоснования и согласования.

3. `Этапы_разработки_Telegram_bot_EnergyCFO.md`
   - основной рабочий файл по этапам реализации;
   - использовать как дорожную карту разработки;
   - после каждого значимого шага обновлять статус, чекбоксы, логи и новые подпункты.

4. `README.md`
   - общий вход в проект;
   - использовать для быстрой ориентации по документам, параметрам MVP и порядку старта.

5. `Чеклист_подготовки_к_разработке.md`
   - использовать для проверки готовности доступов, домена, сервера, Telegram BotFather, Google и репозитория.

6. `Переменные_окружения_env.md`
   - использовать как источник по `.env`, секретам, переменным окружения и правилам безопасности;
   - реальные токены, пароли, OAuth secrets и refresh tokens не записывать в Markdown и не коммитить.

7. `Сценарии_бота_и_тексты.md`
   - использовать как источник пользовательских и админских сценариев, текстов сообщений, кнопок, ошибок и уведомлений.

## Правила работы

1. Перед началом каждой задачи сначала определи, к какому этапу из `Этапы_разработки_Telegram_bot_EnergyCFO.md` она относится.

2. Работай по `Этапы_разработки_Telegram_bot_EnergyCFO.md` как по основному сценарию реализации.

3. Если задача касается требований, сценариев, статусов, бизнес-правил или приемки, опирайся на `ТЗ_Telegram_bot_EnergyCFO_v1.4.md`.

4. Если задача касается архитектуры, библиотек, БД, backend, webhook, Docker, Nginx, SSL или Google API, опирайся на `Стек_технологий_Telegram_bot_EnergyCFO.md`.

5. Если задача касается порядка реализации, контроля прогресса, статусов этапов и логов разработки, опирайся на `Этапы_разработки_Telegram_bot_EnergyCFO.md`.

6. Если задача касается текстов Telegram-бота, кнопок, уведомлений, ошибок и админских сообщений, опирайся на `Сценарии_бота_и_тексты.md`.

7. Если задача касается доступов, домена, BotFather, Google, сервера или подготовки окружения, опирайся на `Чеклист_подготовки_к_разработке.md`.

8. Если задача касается `.env`, токенов, OAuth, БД, секретов и конфигурации, опирайся на `Переменные_окружения_env.md`.

9. После выполнения каждого этапа или подэтапа обязательно обновляй `Этапы_разработки_Telegram_bot_EnergyCFO.md`:
   - отмечай сделанное и несделанное через чекбоксы;
   - фиксируй текущий статус;
   - если появились новые мелкие задачи, добавляй их как подпункты;
   - если этап завершен полностью, явно помечай это в файле.

10. Под каждым выполненным этапом в `Этапы_разработки_Telegram_bot_EnergyCFO.md` кратко записывай лог:
    - что именно сделано;
    - какие файлы созданы или изменены;
    - с какими трудностями столкнулся;
    - почему были приняты конкретные технические решения;
    - какие проверки выполнены.

11. После каждого значимого шага кратко фиксируй прогресс в `Этапы_разработки_Telegram_bot_EnergyCFO.md`.

12. Выполняй задачи максимально самостоятельно. После завершения этапа проводи self-check кода, логики и соответствия ТЗ.

13. Если этап требует участия человека, явно напиши в ответе раздел:

    `Как проверить этот этап вручную`

    В нем дай короткий чек-лист проверки.

14. Проси человека вмешаться только в случаях, когда автоматизация невозможна:
    - создание Telegram-бота в BotFather;
    - ввод или подтверждение секретов;
    - Google OAuth-авторизация владельца;
    - проверка интерфейса в Telegram;
    - финальное согласование перед production-запуском;
    - действия с паролями и приватными доступами.

15. Не меняй утвержденный стек из `Стек_технологий_Telegram_bot_EnergyCFO.md` без явного обоснования. Если предлагаешь замену, сначала укажи:
    - что меняется;
    - зачем это нужно;
    - какие риски;
    - как это влияет на сроки и поддержку.

16. Не раскрывай и не записывай в Markdown:
    - Telegram bot token;
    - Google client secret;
    - Google refresh token;
    - пароли;
    - приватные SSH-ключи;
    - строку подключения к production-БД с паролем.

17. Секреты должны храниться только в `.env` или в настройках окружения на сервере. В Git должен попадать только `.env.example`.

18. Не перезаписывай утвержденные документы без явного подтверждения. Если нужна новая версия документа, создай файл с новой версией или суффиксом.

19. Если задача выходит за рамки текущего этапа, сначала явно укажи это и предложи:
    - выполнить сейчас;
    - перенести в backlog;
    - создать отдельную задачу.

20. Если обнаружено противоречие между файлами:
    - для продукта приоритет у ТЗ;
    - для технологий приоритет у файла стека;
    - для порядка работ приоритет у файла этапов;
    - для текстов бота приоритет у файла сценариев;
    - противоречие зафиксируй в ответе и предложи корректировку.

## Формат ответа при выполнении задач

В каждом ответе придерживайся структуры:

1. `Этап`
   - укажи номер и название этапа из `Этапы_разработки_Telegram_bot_EnergyCFO.md`;
   - если задача относится к нескольким этапам, перечисли их.

2. `Что буду делать`
   - коротко опиши план действий.

3. `Что сделано`
   - после выполнения перечисли созданные и измененные файлы;
   - укажи ключевые решения.

4. `Что обновлено в плане`
   - напиши, какие чекбоксы, статусы или логи внесены в `Этапы_разработки_Telegram_bot_EnergyCFO.md`.

5. `Проверка`
   - укажи автоматические проверки, которые выполнены;
   - если проверки не выполнялись, прямо напиши почему.

6. `Как проверить вручную`
   - добавляй только если нужен ручной контроль.

7. `Git`
   - после завершения этапа обязательно спроси:

     `Нужно ли сделать commit и push на GitHub?`

   - автоматически commit и push не выполнять без подтверждения.

## Контроль версий

1. После завершения этапа спрашивай, нужно ли сделать commit.
2. Commit делать только после подтверждения.
3. Push на GitHub делать только после подтверждения.
4. Перед commit показывать краткий список измененных файлов.
5. Не коммитить `.env`, секреты, бэкапы, логи и временные файлы.
