# Стек технологий Telegram-бота EnergyCFO

Документ фиксирует рекомендуемый технологический стек для MVP Telegram-бота **EnergyCFO | Встречи** и будущего развития проекта: Telegram Mini App, веб-админки, Google Sheets-отчетности и возможной CRM-интеграции.

Основа документа: [ТЗ_Telegram_bot_EnergyCFO_v1.4.md](ТЗ_Telegram_bot_EnergyCFO_v1.4.md).

## 1. Краткое резюме стека

| Уровень | Рекомендуемая технология | Назначение |
|---|---|---|
| Runtime | Node.js LTS | выполнение backend-приложения |
| Язык | TypeScript | строгая типизация и снижение ошибок |
| Telegram-бот | Telegraf | работа с Telegram Bot API |
| HTTP API | Fastify | webhook Telegram, будущий API для Mini App |
| База данных | PostgreSQL | заявки, пользователи, встречи, настройки |
| ORM / миграции | Prisma | схема данных, миграции, типобезопасный доступ |
| Валидация данных | Zod | проверка входных данных и настроек |
| Google-интеграции | Google APIs for Node.js | Google Calendar, Google Meet, OAuth |
| Даты и время | Luxon | МСК, UTC, интервалы, слоты |
| Логи | Pino | структурированные технические логи |
| Фоновые задачи | cron-процесс + PostgreSQL | напоминания, SLA, автоотмена |
| Развертывание | Docker Compose | воспроизводимый запуск на VPS |
| Reverse proxy | Nginx | HTTPS, маршрутизация webhook |
| SSL | Let's Encrypt / Certbot | сертификат для `meet.energycfo.pro` |
| Тестирование | Vitest | unit- и интеграционные тесты |
| Хостинг | Beget Cloud VPS | production-сервер |
| DNS | REG.RU | управление `energycfo.pro` |

## 2. Почему выбран такой стек

| Решение | Почему подходит проекту |
|---|---|
| Node.js + TypeScript | Telegram-бот, webhook, API и фоновые задачи хорошо ложатся на событийную модель Node.js; TypeScript снижает риск ошибок в статусах, датах и данных заявок |
| Telegraf | зрелая библиотека для Telegram-ботов на Node.js, поддерживает кнопки, middleware, webhook и сценарии |
| Fastify | легкий и быстрый HTTP-сервер, удобен для webhook сейчас и Mini App API в будущем |
| PostgreSQL | надежная реляционная база для заявок, статусов, истории, настроек и аналитики |
| Prisma | упрощает работу с PostgreSQL, миграциями и типизацией данных |
| Luxon | уменьшает риск ошибок с часовыми поясами, особенно при хранении UTC и показе МСК |
| Docker Compose | упрощает повторяемый запуск backend, базы и служебных процессов |
| Nginx + SSL | стандартная схема для безопасного публичного webhook Telegram |

## 4. Backend

### Рекомендуемый выбор

| Параметр | Значение |
|---|---|
| Runtime | Node.js LTS |
| Язык | TypeScript |
| HTTP-сервер | Fastify |
| Telegram framework | Telegraf |
| Стиль приложения | модульный монолит |

### Почему модульный монолит

Для MVP не нужна микросервисная архитектура. Проект небольшой, но должен быть разделен на понятные модули:

| Модуль | Назначение |
|---|---|
| `bot` | входящие Telegram-события, кнопки, команды |
| `admin` | админ-действия, согласование, отказ, перенос |
| `calendar` | Google Calendar, Google Meet, OAuth |
| `scheduling` | расчет слотов, ограничения, доступность |
| `notifications` | уведомления пользователю и администратору |
| `jobs` | фоновые задачи, SLA, напоминания |
| `analytics` | базовая статистика |
| `db` | доступ к базе и миграции |
| `config` | настройки и переменные окружения |

### Что не нужно для MVP

| Технология/подход | Почему не нужен сейчас |
|---|---|
| Микросервисы | усложнят разработку и эксплуатацию |
| Kubernetes | избыточен для одного бота |
| Redis | можно обойтись PostgreSQL и фоновой проверкой |
| GraphQL | нет сложного публичного API |
| Очереди уровня RabbitMQ/Kafka | нет высокой нагрузки и массовых событий |

## 5. Telegram-слой

### Рекомендуемая библиотека

| Параметр | Значение |
|---|---|
| Библиотека | Telegraf |
| Режим работы | webhook |
| Endpoint | `https://meet.energycfo.pro/webhook` |
| Админ-доступ | по Telegram ID |

### Почему webhook, а не polling

| Вариант | Оценка |
|---|---|
| Polling | проще на локальном тесте, но хуже для production |
| Webhook | лучше для production, быстрее реакция, меньше лишних запросов |

Для локальной разработки можно временно использовать polling или туннель, но production должен работать через webhook.

### Что должно поддерживаться

| Функция | Назначение |
|---|---|
| Inline-кнопки | выбор темы, даты, слота, действий администратора |
| Reply-кнопки | основные разделы меню |
| Callback-обработка | действия по кнопкам без лишнего текста |
| FSM/состояния | пошаговый сбор данных пользователя |
| Защита админ-команд | скрыть админ-функции от обычных пользователей |
| Обработка ошибок | понятные сообщения без технических деталей |

## 6. База данных

### Рекомендуемый выбор

| Параметр | Значение |
|---|---|
| СУБД | PostgreSQL |
| Размещение MVP | на Beget VPS |
| Дальнейшее масштабирование | Beget Cloud PostgreSQL |
| ORM | Prisma |
| Миграции | Prisma Migrate |

### Почему PostgreSQL

| Причина | Польза |
|---|---|
| Реляционная модель | удобно хранить заявки, пользователей, встречи и статусы |
| Транзакции | важно при согласовании, переносе и отмене |
| Индексы | быстрый поиск заявок, дат, статусов |
| Расширяемость | можно добавить отчетность, Mini App, веб-админку |
| Надежность | подходит для production даже на небольшом VPS |

### Основные таблицы

| Таблица | Что хранит |
|---|---|
| `users` | Telegram ID, ФИО, компания, должность, email |
| `contact_preferences` | предпочтительный канал связи и доп. контакт |
| `meeting_requests` | заявки на встречу |
| `meetings` | согласованные встречи |
| `availability_settings` | рабочие часы, лаг, буфер, лимиты |
| `excluded_dates` | даты, закрытые администратором |
| `status_history` | история статусов заявки |
| `google_tokens` | OAuth-токены Google |
| `notification_log` | отправленные уведомления |
| `system_logs` | технические ошибки и события |

### Важные технические правила

| Правило | Требование |
|---|---|
| Время в БД | хранить в UTC |
| Показ времени | отображать в МСК |
| Статусы | хранить как контролируемый список |
| Email | индексировать для поиска |
| Telegram ID | уникальный идентификатор пользователя |
| Google event ID | хранить для обновления и отмены события |
| История изменений | не затирать важные статусы |

## 8. Расчет слотов и работа со временем

### Рекомендуемая библиотека

| Параметр | Значение |
|---|---|
| Библиотека | Luxon |
| Хранение времени | UTC |
| Отображение | МСК |
| Пользовательский часовой пояс | в MVP не используется |

### Почему важно отдельно выделить модуль слотов

Расчет доступного времени - ключевая логика проекта. Ее нельзя размазывать по Telegram-сценариям или Google-интеграции.

Модуль слотов должен учитывать:

| Ограничение | Значение |
|---|---|
| Рабочие дни | Пн-Пт |
| Рабочие часы | 10:00-18:00 МСК |
| Длительность | 60 минут |
| Буфер | 30 минут |
| Лаг | 12 часов |
| Горизонт пользователя | 2 месяца |
| Горизонт администратора | 3 месяца |
| Лимит встреч | 5 в день |
| Исключенные даты | закрывают день полностью |
| Занятость календаря | исключает занятые интервалы |

### Контрольные сценарии для тестов

| Сценарий | Ожидаемый результат |
|---|---|
| Сегодня 09:00, слот сегодня 10:00 | не показывается из-за лага 12 часов |
| День исключен администратором | не показывается |
| В календаре встреча 11:00-12:00 | слот пересечения не показывается |
| Встреча заканчивается в 12:00 | следующий слот не раньше 12:30 |
| Уже 5 встреч в день | новые слоты на день не показываются |

## 9. Фоновые задачи

### Рекомендуемый подход для MVP

| Параметр | Значение |
|---|---|
| Механизм | cron-процесс внутри backend или отдельный worker |
| Хранилище состояния | PostgreSQL |
| Частота проверки | каждые 10-15 минут |
| Отдельный Redis | не нужен для MVP |

### Почему без Redis

Для MVP нагрузка небольшая, а задачи простые: напоминания, SLA, автоотмена, завершение встреч. Их можно надежно выполнять через периодическую проверку PostgreSQL и журнал отправленных уведомлений.

### Задачи

| Задача | Периодичность |
|---|---|
| Напоминание администратору через 2 часа | каждые 10-15 минут |
| Напоминание администратору через 12 часов | каждые 10-15 минут |
| Напоминание администратору через 24 часа | каждые 10-15 минут |
| Напоминание пользователю за 24 часа | каждые 10-15 минут |
| Напоминание пользователю за 1 час | каждые 10-15 минут |
| Автоотмена несогласованных заявок ближе 12 часов | каждые 10-15 минут |
| Завершение прошедших встреч | раз в час |

### Защита от дублей

| Правило | Зачем нужно |
|---|---|
| Хранить факт отправки уведомления | чтобы не отправлять одно и то же сообщение много раз |
| Делать задачи идемпотентными | повторный запуск не должен ломать данные |
| Фиксировать ошибки | чтобы понимать, какие уведомления не ушли |

## 10. Развертывание

### Рекомендуемый подход

| Компонент | Решение |
|---|---|
| Сервер | Beget Cloud VPS `energycfo` |
| ОС | Ubuntu |
| Контейнеризация | Docker Compose |
| Reverse proxy | Nginx |
| SSL | Let's Encrypt / Certbot |
| Домен | `energycfo.pro` |
| Поддомен | `meet.energycfo.pro` |

### Почему Docker Compose

| Причина | Польза |
|---|---|
| Повторяемость | проще поднять одинаковую среду |
| Изоляция | backend и БД не смешиваются с системными пакетами |
| Обновления | можно обновлять приложение контролируемо |
| Переносимость | проще переехать на другой VPS |

### Возможная структура контейнеров

| Контейнер | Назначение |
|---|---|
| `app` | Telegram-бот и HTTP API |
| `worker` | фоновые задачи |
| `postgres` | база данных MVP |

Nginx может быть установлен на хосте или тоже запущен в контейнере. Для новичкового сопровождения чаще проще держать Nginx на хосте, а приложение и БД - в Docker Compose.

### Важный нюанс по текущему серверу

На домене уже есть поддомены `n8n.energycfo.pro` и `supabase.energycfo.pro`, которые указывают на тот же IP `45.144.179.251`. Перед развертыванием нужно проверить, какие сервисы уже работают на сервере и какие порты заняты.

Правило: не использовать внутреннюю базу Supabase для Telegram-бота без отдельного осознанного решения. Для MVP безопаснее отдельная база PostgreSQL или отдельная схема, чтобы не смешивать данные разных проектов.

## 14. Резервное копирование

### Рекомендуемая схема MVP

| Параметр | Значение |
|---|---|
| Тип | ежедневный `pg_dump` |
| Хранение | 14 дней |
| Место хранения | сервер + по возможности внешнее хранилище |
| Проверка восстановления | минимум один раз до запуска |

### Что нужно включить в бэкап

| Данные | Нужно ли сохранять |
|---|---|
| PostgreSQL | да |
| `.env` production | да, отдельно и защищенно |
| Код | хранится в Git |
| Логи | не обязательно, только при расследовании |

### Что важно контролировать

| Риск | Контроль |
|---|---|
| Бэкапы забьют диск | удалять старше 14 дней |
| Бэкап поврежден | периодически проверять восстановление |
| Бэкап хранится только на сервере | добавить внешнее хранение при развитии проекта |

| `docker-compose.yml` | запуск сервисов |
| `prisma/` или `migrations/` | схема БД |

### Что не должно попасть в Git

| Файл/данные | Почему |
|---|---|
| `.env` | секреты |
| Бэкапы БД | персональные данные |
| Логи production | лишние данные |
| Token Telegram | управление ботом |
| Google OAuth secret | доступ к календарю |

## 18. Альтернативы и почему они не выбраны как основные

| Альтернатива | Почему не основной выбор |
|---|---|
| Python + aiogram | хороший вариант, но для будущего Mini App и веб-API TypeScript-стек будет единообразнее |
| no-code / n8n | можно быстро собрать прототип, но сложнее контролировать бизнес-правила слотов и статусы |
| Supabase как основная backend-платформа | полезен, но MVP проще держать в собственном backend без привязки логики к BaaS |
| SQLite | мало подходит для production с фоновыми задачами и ростом функций |
| MongoDB | слабее подходит для статусов, связей, отчетности и транзакций |
| Redis + BullMQ | надежно для очередей, но добавляет лишний сервис для MVP |
| NestJS | мощный framework, но для текущего MVP может быть избыточен |

## 21. Стек для будущего Mini App

Mini App не входит в MVP, но архитектура должна быть готова к нему.

### Рекомендуемый будущий стек

| Компонент | Рекомендация |
|---|---|
| Frontend | React + TypeScript |
| Сборка | Vite |
| UI | собственные компоненты или легкая UI-библиотека |
| API | Fastify endpoints текущего backend |
| Авторизация | Telegram Mini App init data |
| Даты | Luxon |
| Хостинг | тот же `meet.energycfo.pro`, отдельный путь `/app` |

### Что нужно предусмотреть уже в MVP

| Требование | Почему |
|---|---|
| API-слой не смешивать с Telegram-сценариями | Mini App будет обращаться к backend напрямую |
| Бизнес-логику слотов держать отдельно | Mini App и бот должны считать одинаково |
| Авторизацию пользователя выделить отдельно | Telegram-бот и Mini App используют разные входы |
| Настройки хранить в БД | Mini App сможет их читать и менять |

## 22. Риски выбранного стека

| Риск | Как снизить |
|---|---|
| На VPS уже есть другие сервисы | проверить занятые порты и потребление RAM |
| PostgreSQL на том же VPS увеличит нагрузку | контролировать RAM и диск, при росте вынести БД |
| OAuth Google может потребовать настройки consent screen | заложить отдельный этап подключения |
| Webhook требует корректного SSL | настроить Nginx и Certbot до production |
| Фоновые задачи могут отправить дубли | хранить журнал отправленных уведомлений |
| Ошибки часовых поясов | хранить UTC, показывать МСК, покрыть тестами |

## 23. Итоговая рекомендация

Для MVP использовать стек:

```text
Node.js LTS + TypeScript
Telegraf
Fastify
PostgreSQL
Prisma
Luxon
Google APIs for Node.js
Pino
Docker Compose
Nginx + Let's Encrypt
Beget Cloud VPS
```

Этот стек достаточно простой для пошаговой совместной разработки, но при этом не закрывает путь к Mini App, веб-админке, Google Sheets-отчетности и CRM-интеграции.
